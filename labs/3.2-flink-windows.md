# Demo 3.2 — Implementing Windowed Aggregations with Watermarks in Flink

**Session:** 3.2 — Advanced Flink: Windows, State & Guarantees  
**Format:** Instructor-led hands-on demo  
**Tools:** Docker Compose, Kafka, Flink SQL Client, Flink Web UI

## Objective

Show how Flink handles time, windows, and late data. Make the abstract concepts of watermarks, state backends, and checkpointing concrete with a live pipeline.

## Setup

- Kafka + Flink cluster from Demo 3.1 (reused)
- A producer that emits events with embedded timestamps (event time), some intentionally delayed
- Flink SQL client for interactive windowed queries

## Walkthrough

### Step 1 — Tumbling window

Count events per `user_id` in 1-minute tumbling windows (Flink SQL):
- `SELECT window_start, user_id, COUNT(*) FROM TABLE(TUMBLE(...)) GROUP BY ...`
- Observe windows opening, accumulating, and closing
- Show: results emit only when the watermark advances past the window end

### Step 2 — Event time vs processing time

Pause the producer for 30 seconds, then send a burst:
- Processing time: the burst lands in a single window
- Event time: events are assigned to their original windows
- Show both side by side — the results are completely different

### Step 3 — Late data and watermarks

Configure a watermark with 10-second allowed lateness:
- Send events that arrive 5 seconds late → included in the correct window
- Send events that arrive 15 seconds late → dropped (past the watermark)
- Show the watermark advancing in the Flink Web UI (watermark metrics)
- Discuss: how to choose watermark delay (latency vs completeness)

### Step 4 — State inspection

While a stateful query runs:
- Show RocksDB state backend in the Flink UI: state size per operator
- Observe state growth over time as more windows are tracked
- Trigger a savepoint — inspect the savepoint directory
- Restart from savepoint — show state is fully recovered

### Step 5 — Session windows

Switch to session windows with a 2-minute inactivity gap:
- Show sessions of varying length depending on user activity
- A user active for 10 minutes gets a single long window
- Intermittent activity produces multiple short windows
- Show: session windows merge in Flink — observe state size implications

## Key Takeaways

- Event time is almost always what you want — processing time misleads when there's any delay
- Watermarks are Flink's mechanism for deciding "when is a window complete enough to emit?"
- State backends (RocksDB) allow Flink to handle state larger than memory
- Savepoints enable zero-downtime upgrades and pipeline migrations
- Session windows are the most expressive but the most expensive in terms of state
