# Demo 4.1 — Probabilistic Data Structures: Exact vs Approximate

**Session:** 4.1 — Probabilistic Data Structures  
**Format:** Instructor-led hands-on demo  
**Tools:** Python, ClickHouse (from Day 3 setup)

## Objective

Build three probabilistic data structures from scratch in Python, run them against real data, and compare accuracy vs memory vs speed against exact computation. Then show their built-in counterparts in ClickHouse.

## Setup

- Python with only standard library + `hashlib` (no probabilistic libraries — the point is to build them)
- A dataset with known exact answers for ground-truth comparison: the GitHub events from Day 1 work well (distinct users, event frequencies, response time percentiles)
- ClickHouse still running from Demo 3.4

## Walkthrough

### Step 1 — HyperLogLog: counting distinct users

Implement a minimal HyperLogLog:
- Hash each username, use the first `p` bits as a bucket index, count leading zeros in the rest
- Start with `p=4` (16 buckets) — show the estimate is noisy but in the right ballpark
- Increase to `p=14` (16K buckets, ~12 KB) — estimate is within 1–2% of exact
- Compare: `set()` of all usernames vs the 12 KB HLL register array
- Show memory usage side by side: exact set grows with cardinality, HLL stays fixed
- Then run `SELECT uniqHLL12(user)` in ClickHouse — same algorithm, same accuracy, zero effort

### Step 2 — Bloom filter: has this repo been seen?

Implement a Bloom filter:
- Allocate a bit array of size `m`, use `k` hash functions
- Insert all repository names from the dataset
- Test membership: zero false negatives, measurable false positive rate
- Vary `m` and `k` — plot false positive rate vs memory
- Show the practical use case: a Flink operator that needs to deduplicate events without keeping the full set in state

### Step 3 — Count-Min Sketch: event frequency estimation

Implement a Count-Min Sketch:
- `d` rows × `w` columns, each row uses a different hash function
- Insert event counts, query frequency estimates
- Compare against an exact `Counter`: show that CMS overestimates but never underestimates
- Show heavy hitters: top-K frequent items using CMS vs exact sort
- Demonstrate the space/accuracy trade-off by varying `w`

### Step 4 — Combining them in a streaming scenario

Simulate a streaming pipeline that processes events one at a time:
- HLL tracks distinct users seen so far
- Bloom filter deduplicates events (skip if already processed)
- CMS estimates per-repo event frequency for top-K
- Total memory: a few hundred KB. Equivalent exact state: hundreds of MB.
- Show how the error bounds compose: each structure has independent guarantees

### Step 5 — ClickHouse built-ins

Run the equivalent queries in ClickHouse on the billion-row table from Demo 3.4:
- `uniqHLL12()` — HyperLogLog cardinality
- `quantileTDigest()` — approximate percentiles via t-digest
- Compare `uniq()` (exact) vs `uniqHLL12()` (approximate): speed difference on 1B rows
- Show `EXPLAIN` output: ClickHouse uses these structures internally for optimization

## Key Takeaways

- Probabilistic structures give bounded-error answers in fixed memory — they don't grow with data size
- HyperLogLog: ~12 KB for cardinality estimates within 1–2% on billions of elements
- Bloom filters: definite "no" answers in constant space — useful for deduplication and pre-filtering
- Count-Min Sketch: frequency estimation that only overestimates — safe for heavy-hitter detection
- These aren't academic curiosities: ClickHouse, Spark, and Flink use them internally and expose them as functions
