# Demo 3.4 — Modeling and Querying a Billion-Row Analytics Dataset in ClickHouse

**Session:** 3.4 — ClickHouse: Real-Time Analytics at Scale  
**Format:** Instructor-led hands-on demo  
**Tools:** Docker, ClickHouse server, `clickhouse-client`

## Objective

Build a complete analytics workflow in ClickHouse: ingest data, choose the right engine, write analytical queries, and use materialized views for real-time pre-aggregation. Show why ClickHouse is fast by observing its behavior.

## Setup

- ClickHouse single-node in Docker
- Dataset: a large event log (~100M–1B rows), either synthetic or from a public dataset (e.g., GitHub events, web analytics)
- Data pre-staged as CSV or Parquet for fast ingestion

## Walkthrough

### Step 1 — Ingestion speed

Bulk-load 100M rows from Parquet:
- Show the ingestion throughput (rows/s, MB/s)
- Compare with inserting the same data into PostgreSQL
- Explain: ClickHouse writes data in sorted, compressed column chunks — optimized for append

### Step 2 — MergeTree internals

Inspect what ClickHouse wrote to disk:
- Show the parts directory: each part is a sorted, immutable chunk
- Column files, primary index (sparse), skip indexes
- Trigger a manual merge (`OPTIMIZE TABLE`) — show parts consolidating
- Explain: background merges keep the number of parts manageable

### Step 3 — Analytical queries

Run progressively complex queries:
- Count by day → instant (primary key scan)
- Top 10 users by event count → fast (columnar scan + sort)
- Percentile of response times → use `quantile()` (approximate, fast)
- Funnel analysis → multi-step aggregation

Show `EXPLAIN` and `system.query_log` for each:
- Rows read, bytes read, time
- Which granules were scipped via the primary key

### Step 4 — Wrong ORDER BY

Create the same table with a different `ORDER BY`:
- Original: `ORDER BY (date, user_id)` — date-range queries are fast
- Alternative: `ORDER BY (user_id, date)` — date-range queries now scan everything
- Show the query time difference (10–100x)
- Key lesson: `ORDER BY` in ClickHouse is the primary index, not a sorting preference

### Step 5 — Materialized views

Create a materialized view that pre-aggregates events into hourly counts:
- Insert new rows into the source table
- Query the materialized view — results are already computed
- Show: the materialized view is incrementally updated on each insert, not recomputed
- Compare query time: source table vs materialized view (orders of magnitude)

### Step 6 — Projections

Add a projection with a different sort order:
- Original table sorted by `(date, user_id)`, projection sorted by `(user_id, date)`
- ClickHouse automatically picks the best sort order per query
- Show via `EXPLAIN` that different queries hit different projections

## Key Takeaways

- ClickHouse achieves speed through sorted columnar storage, vectorized execution, and aggressive compression
- `ORDER BY` is the most important schema decision — it determines which queries are fast
- Materialized views provide real-time pre-aggregation with no manual refresh
- Projections give you multiple sort orders without duplicating tables
- ClickHouse is not a general-purpose database — it trades update/delete capability for read performance
